# PRD: 난세영걸전 남은 작업 정리

## 1. Introduction / Overview

이 문서는 [기존 PRD](/Users/jangmyeong-gyun/ralphthon/tasks/prd-nanse-yeonggeoljeon.md)를 기준으로, `2026-03-29` 현재 저장소에 이미 구현된 수직 슬라이스 베이스라인 이후에 반드시 남아 있는 작업만 별도 범위로 정리한 후속 PRD다.

현재 프로젝트는 다음 요소를 이미 갖추고 있다.

- `Flutter` 기반 앱 부트스트랩과 `Web` 직접 라우트 진입
- `SC-01 ~ SC-14` 화면 셸
- 10개 스테이지/5명 핵심 장수 데이터
- 헤드리스 `BattleEngine`
- 수동 `이동`, `공격`, `책략`, `도구`, `대기`, `턴 종료`
- `flutter analyze`, `flutter test` 기반 회귀 검증

하지만 현재 구현은 “완성된 캠페인 SRPG”가 아니라 “핵심 구조를 갖춘 수직 슬라이스”에 가깝다. 따라서 본 문서의 목적은 남은 구현을 우선순위가 있는 작업 단위로 재정리해, 이후 실행 시 문서와 실제 완료 기준이 어긋나지 않게 만드는 것이다.

## 2. Goals

- 스테이지 메타데이터 수준에 머물러 있는 캠페인 규칙을 실제 플레이 가능 규칙으로 승격한다.
- 전투 코어가 스테이지 목표, 이벤트, 일기토, 저장 상태, 성장 상태를 실제로 처리하도록 확장한다.
- 화면 셸 수준인 UI를 실제 게임 데이터와 연결해 목업이 아닌 기능 화면으로 전환한다.
- `Playwright` 기반 웹 시각 검증 체인을 추가해 PRD의 완료 기준과 자동화 품질 게이트를 일치시킨다.
- PixelLab/Gemini/ElevenLabs 기반 자산 파이프라인을 실제 프로젝트 자산 구조와 연결한다.
- 휴리스틱 AI와 반복 시뮬레이션을 도입해 밸런스 목표를 측정 가능한 형태로 바꾼다.

## 3. Non-Goals

- 삼국 전체를 아우르는 대규모 분기 캠페인 확장
- 신규 플레이어블 장수 추가
- 온라인 기능, PvP, 라이브 서비스 기능
- 3D 전장 전환
- 풀보이스 컷신

## 4. Current Baseline Summary

### 4.1 Already Implemented

- 앱은 `Flutter` 단일 코드베이스로 동작하며 웹 라우트 직접 진입이 가능하다.
- 14개 필수 화면에 대응하는 경로가 존재한다.
- 10개 스테이지 데이터와 5명 핵심 장수 데이터가 정적 저장소에 정리돼 있다.
- 전투 코어는 격자 이동, 사거리 판정, 피해 계산, 턴 진행, 헤드리스 시뮬레이션을 제공한다.
- 전투 HUD는 수동 커맨드 입력과 기본 상태 표시를 제공한다.
- `flutter analyze`, `flutter test`는 통과 중이다.

### 4.2 Gaps This Document Covers

- 스테이지 목표/패배 조건/이벤트가 실제 규칙으로 실행되지 않음
- 일기토, 보상, 성장, 저장/불러오기, 출전 편성이 대부분 정적 목업 상태
- Playwright 순회 및 시각 검증 자동화 부재
- 실제 스프라이트/타일맵/오디오 자산 통합 부재
- 휴리스틱 AI와 밸런스 시뮬레이션이 요구 수준에 미달

## 5. Product Pillars For Remaining Work

### Pillar 1: 메타데이터를 실제 전투 규칙으로 승격

- 각 스테이지의 승리 조건, 패배 조건, 이벤트 트리거를 코어 규칙으로 실행한다.
- 전투 결과가 단순 보스 처치가 아니라 스테이지 고유 목표 달성으로 결정되게 만든다.

### Pillar 2: 목업 화면을 실제 게임 흐름으로 연결

- 브리핑, 편성, 전투, 결과, 장수 관리, 저장/불러오기가 동일 상태 모델을 공유해야 한다.
- UI 문구와 더미 카드가 아니라 실제 게임 상태가 화면에 반영돼야 한다.

### Pillar 3: 검증 가능한 품질 게이트 확립

- 코어 회귀 테스트와 함께 웹 화면 순회 검증을 자동화한다.
- “보여 보이는 것”이 아니라 실제 렌더링을 기준으로 완료를 판정한다.

## 6. Remaining Workstreams / User Stories

### US-001: 스테이지별 승리/패배 규칙 실행

**Description:** 개발자로서 나는 각 스테이지의 승리 조건과 패배 조건을 문자열이 아니라 실제 코어 규칙으로 처리하고 싶다. 그래야 스테이지별 플레이 차이가 전투 결과에 직접 반영된다.

**Acceptance Criteria:**

- [ ] `StageDefinition` 또는 동등 구조가 `objectiveType`, `lossTriggers`, `eventTriggers` 같은 실행 가능한 필드를 가진다.
- [ ] 최소 다음 목표 타입이 코어에서 판정된다: `boss_defeat`, `escape`, `escort`, `hold_position`, `capture_points`.
- [ ] 최소 다음 패배 타입이 코어에서 판정된다: `lord_dead`, `npc_dead`, `turn_limit`, `escape_failure`.
- [ ] 스테이지 1, 4, 6, 8, 10은 서로 다른 승리 조건을 실제로 가진다.
- [ ] 전투 결과 판정이 “보스 생존 여부” 하나에만 의존하지 않는다.
- [ ] 대표 스테이지별 코어 테스트가 추가된다.

### US-002: 스크립트 이벤트와 일기토 구현

**Description:** 플레이어로서 나는 특정 조합과 상황에서 일기토와 중간 이벤트가 발동해 원작 감성을 느끼고 싶다.

**Acceptance Criteria:**

- [ ] 전투 시작 이벤트, 전투 중 이벤트, 종료 이벤트를 공통 이벤트 시스템으로 표현할 수 있다.
- [ ] 최소 4개의 일기토 이벤트가 데이터로 정의된다.
- [ ] 일기토는 특정 조합과 위치/거리/턴 조건을 만족할 때만 발동한다.
- [ ] 일기토 결과는 최소 1개 이상의 실효 보상으로 연결된다: 경험치, 사기, 아이템, 이벤트 분기.
- [ ] 스테이지 2 관우 vs 화웅 일기토가 대표 구현으로 동작한다.
- [ ] 이벤트 발동 여부가 전투 로그 또는 결과 데이터에 기록된다.

### US-003: 편성, 성장, 장비, 보상 시스템 연결

**Description:** 플레이어로서 나는 전투 전후 준비와 성장 결과가 실제 데이터로 반영되길 원한다.

**Acceptance Criteria:**

- [ ] 장수 데이터가 레벨, 경험치, 장비 슬롯, 소모품 슬롯, 전용 기술 보유 여부를 표현한다.
- [ ] 편성 화면에서 출전 인원을 조정할 수 있다.
- [ ] 전투 결과 화면이 경험치, 획득 아이템, 다음 스테이지 해금 상태를 표시한다.
- [ ] 장수 관리 화면이 현재 성장 상태를 실제 데이터에서 읽어 표시한다.
- [ ] 저장 데이터가 편성 상태, 성장 상태, 캠페인 진행도를 보존한다.
- [ ] 조운/제갈량 합류 또는 가용 시점 변화가 캠페인 진행 상태에 따라 반영된다.

### US-004: 저장/불러오기 실구현

**Description:** 플레이어로서 나는 진행 상태를 저장하고 다시 이어서 플레이하고 싶다.

**Acceptance Criteria:**

- [ ] 저장 슬롯 구조가 정의된다.
- [ ] 저장 데이터는 현재 스테이지, 턴, 유닛 상태, 편성, 성장, 획득 아이템을 포함한다.
- [ ] 불러오기 시 동일 상태를 재현할 수 있다.
- [ ] 저장/불러오기 화면이 정적 목업이 아니라 실제 슬롯 목록을 렌더링한다.
- [ ] 직렬화/역직렬화 테스트가 추가된다.

### US-005: 휴리스틱 AI와 밸런스 시뮬레이션 확장

**Description:** 밸런스 디자이너로서 나는 양 진영 공용 휴리스틱 AI를 이용해 스테이지 승률을 반복 측정하고 싶다.

**Acceptance Criteria:**

- [ ] 플레이어측과 적측에 모두 적용 가능한 휴리스틱 정책이 구현된다.
- [ ] 평가 함수가 목표 기여도, 킬 가능성, 반격 위험, 지형 선호, 저체력 생존성, 보스 가치까지 고려한다.
- [ ] 동점 해소와 경미한 이동 변주를 위한 통제된 랜덤성이 도입된다.
- [ ] 고정 시드 사용 시 결과가 재현 가능하다.
- [ ] 스테이지별 반복 시뮬레이션 진입점이 추가된다.
- [ ] 스테이지 `1~3`은 목표 승률 `90%`, `4~10`은 `30%`를 리포트에 표시한다.
- [ ] 시뮬레이션 결과를 저장하거나 비교 가능한 출력 포맷이 정의된다.

### US-006: 실제 전장 자산과 애니메이션 통합

**Description:** 아트 제작자와 플레이어로서 나는 텍스트 상태나 이니셜 아바타가 아니라 실제 삼국지 자산을 화면에서 보고 싶다.

**Acceptance Criteria:**

- [ ] 캐릭터 스프라이트 자산 경로와 로딩 규칙이 정의된다.
- [ ] 각 핵심 장수의 `idle`, `walk`, `attack`, `hit` 애니메이션 자산 스펙이 프로젝트 구조에 반영된다.
- [ ] 전투 HUD는 문자열 상태가 아니라 실제 `idle`/`attack` 프레임 또는 애니메이션을 표시한다.
- [ ] 장수 관리/편성/미니 패널은 각 장수의 `south` 프레임 아이콘을 재사용한다.
- [ ] 전장 타일맵은 단색 박스가 아니라 타일 기반 렌더링으로 대체된다.
- [ ] `pubspec.yaml`에 필요한 자산 선언이 추가된다.

### US-007: 화면별 Playwright 순회 검증

**Description:** 개발자로서 나는 각 화면을 실제 브라우저에서 자동으로 순회 검증하고 싶다. 그래야 목업 단계에서 놓친 레이아웃 오류를 릴리즈 전에 차단할 수 있다.

**Acceptance Criteria:**

- [ ] 웹 실행용 테스트 절차가 문서화된다.
- [ ] `Playwright` 테스트 디렉터리와 실행 스크립트가 추가된다.
- [ ] 필수 화면 14개 경로를 순차 방문하는 테스트가 존재한다.
- [ ] 핵심 UI 존재 여부와 레이아웃 안정성 검사가 포함된다.
- [ ] 데스크톱 해상도와 최소 1개 모바일 해상도 검사가 포함된다.
- [ ] 실패 시 스크린샷 또는 리포트가 남는다.

### US-008: 공통 UI 컴포넌트와 화면 분리

**Description:** 개발자로서 나는 지금 한 파일에 몰려 있는 화면 코드를 작업 단위별로 분리해 이후 기능 추가 시 결합도를 낮추고 싶다.

**Acceptance Criteria:**

- [ ] 화면 구현이 전투, 캠페인, 관리, 설정 등 의미 있는 단위로 분리된다.
- [ ] 공통 패널, 버튼, 팩트 칩, 장수 카드, 타일맵 셀 등의 재사용 컴포넌트 위치가 명확해진다.
- [ ] 화면 모듈 분리 후에도 기존 경로 테스트가 유지된다.
- [ ] 분리 과정에서 코어 규칙이 UI 계층으로 역류하지 않는다.

## 7. Functional Requirements

- `FR-1`: 스테이지 목표와 패배 조건은 실행 가능한 데이터 구조로 표현돼야 한다.
- `FR-2`: 전투 엔진은 스테이지별 목표 타입을 판정해야 한다.
- `FR-3`: 전투 엔진은 이벤트와 일기토 트리거를 처리해야 한다.
- `FR-4`: 일기토는 결과 보상 또는 분기 값을 반환해야 한다.
- `FR-5`: 장수 데이터는 경험치, 장비, 소모품, 전용 기술 상태를 표현해야 한다.
- `FR-6`: 편성 화면은 실제 출전 인원 조정 기능을 제공해야 한다.
- `FR-7`: 결과 화면은 경험치, 아이템, 해금 정보를 실제 데이터에서 표시해야 한다.
- `FR-8`: 저장 데이터는 전투 상태와 캠페인 진행도를 직렬화 가능해야 한다.
- `FR-9`: 저장/불러오기 화면은 실제 슬롯 목록과 입출력 기능을 제공해야 한다.
- `FR-10`: 휴리스틱 AI는 양 진영 모두에 적용 가능해야 한다.
- `FR-11`: 휴리스틱 AI는 제한된 랜덤성을 포함해야 한다.
- `FR-12`: 반복 시뮬레이션은 스테이지 승률과 주요 지표를 산출해야 한다.
- `FR-13`: 전투 화면은 실제 스프라이트 기반 `idle`/`attack` 상태를 표시해야 한다.
- `FR-14`: 관리/편성 화면은 실제 `south` 프레임 아이콘을 재사용해야 한다.
- `FR-15`: 전장 렌더링은 타일맵 자산을 사용해야 한다.
- `FR-16`: Playwright 테스트는 필수 화면 전체를 순회해야 한다.
- `FR-17`: Playwright 테스트는 데스크톱과 모바일 뷰포트를 모두 포함해야 한다.
- `FR-18`: UI 코드 분리 이후에도 기존 공통 레이아웃 규칙은 유지돼야 한다.

## 8. Technical Considerations

### 8.1 Core/Data Separation

- `domain`은 계속 순수 계산 계층을 유지해야 한다.
- 스테이지 목표, 이벤트, 저장 데이터는 모두 UI 독립적인 모델이어야 한다.
- “현재 상태 + 명령 -> 다음 상태” 구조를 깨지 않도록 이벤트 처리도 상태 전이형으로 설계한다.

### 8.2 Save Schema

- 최소 저장 단위:
  - 캠페인 진행 단계
  - 현재 스테이지 ID
  - 현재 턴
  - 아군/적군 유닛 상태
  - 장수 성장 상태
  - 장비/아이템 보유 상태
  - 해금/클리어 상태

### 8.3 Playwright Tooling

- 권장 위치: `tooling/playwright/`
- 권장 내용:
  - 라우트 순회 스크립트
  - 뷰포트 세트업
  - 필수 요소 확인
  - 실패 스크린샷 저장
- `flutter run -d web-server` 또는 동등 웹 실행 흐름과 결합 가능한 형태여야 한다.

### 8.4 Asset Integration

- 캐릭터/타일/초상화/오디오는 `assets/` 하위 규칙을 먼저 정한 뒤 연결한다.
- 실제 자산 반영 전에도 메타데이터 파일 또는 매니페스트를 먼저 두어 UI가 문자열 하드코딩에 머물지 않게 한다.

## 9. Suggested Delivery Order

1. 스테이지 목표/패배 조건 데이터 구조화
2. 코어 이벤트/일기토 시스템 구현
3. 결과/성장/저장 스키마 정의
4. 편성/저장/결과 화면 실제 데이터 연결
5. 휴리스틱 AI 개선 및 반복 시뮬레이션 도입
6. 자산 매니페스트와 스프라이트/타일맵 로딩 연결
7. Playwright 시각 검증 체인 추가
8. 화면 파일 분리와 공통 컴포넌트 정리

### 9.1 Execution Discipline

- 위 작업 순서에서 하나의 단계가 완료될 때마다 즉시 로컬 커밋을 생성한다.
- 각 커밋은 저장소의 Lore Commit Protocol을 따른다.
- 각 단계 완료 커밋 직후 원격 저장소에 `push`까지 수행한다.
- 여러 단계를 한 커밋으로 묶지 않고, 단계 단위의 검증 가능한 커밋/푸시 흐름을 유지한다.
- 커밋 전 최소 검증은 해당 단계와 직접 관련된 테스트 또는 확인 절차를 포함해야 한다.

## 10. Verification & Quality Gates

- `flutter analyze` 통과
- `flutter test` 통과
- 대표 스테이지 코어 회귀 테스트 통과
- 저장/불러오기 직렬화 테스트 통과
- 밸런스 시뮬레이션 리포트 생성 성공
- `Playwright` 전체 화면 경로 순회 통과
- 모바일/데스크톱 뷰포트 모두 치명적 레이아웃 오류 0건
- 각 완료 단계마다 Lore Commit Protocol 형식 커밋 생성 및 원격 `push` 완료

## 11. Success Metrics

- 스테이지 결과가 단순 보스 처치 외 목표 타입으로 실제 판정된다.
- 최소 4개의 일기토가 실제 발동 가능하다.
- 저장/불러오기 후 상태 재현 불일치가 0건이다.
- 스프라이트/타일맵/아이콘이 실제 자산 기반으로 렌더링된다.
- `Playwright` 기준 필수 화면 전부가 순회 검증을 통과한다.
- 반복 시뮬레이션 결과가 각 스테이지 목표 승률을 추적 가능하게 만든다.
