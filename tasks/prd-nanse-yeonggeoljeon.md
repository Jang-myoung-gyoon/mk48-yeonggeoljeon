# PRD: 난세영걸전

## 1. Introduction / Overview

`난세영걸전`은 `삼국지 영걸전`의 장점인 유비 중심 서사, 스테이지별 승리 조건, 턴제 전술 전투, 일기토 연출을 현대적으로 재구성한 SRPG다. 플레이어는 유비 세력을 이끌며 `도원결의`부터 `형주 진입`까지 이어지는 10개 스테이지를 클리어하고, 5명의 핵심 장수를 성장시키며 삼국지 특유의 의리, 책략, 지형 활용 전투를 경험한다.

본 프로젝트의 핵심 목표는 원작 감성의 단순 복제가 아니라, `삼국지 영걸전`이 보여준 "역사 서사 + 미션형 전투 + 장수 성장" 구조를 짧고 응집된 캠페인으로 재해석하는 것이다. 게임 구현은 `Flutter`를 사용하며, 1차 타겟 플랫폼은 `Web`과 `Android`다. 그래픽과 리소스 제작은 `PixelLab`, `Gemini 이미지 MCP`, `Gemini BGM`, `ElevenLabs`를 중심으로 설계하며, 모든 비주얼과 음향이 후한 말~삼국 시대 분위기를 일관되게 반영해야 한다.

## 2. Goals

- `삼국지 영걸전` 감성의 SRPG 전투 구조를 10개 스테이지 분량의 완결형 캠페인으로 제공한다.
- 유비, 관우, 장비, 조운, 제갈량 5명의 개성과 역할이 전투 시스템에서 분명하게 드러나도록 한다.
- 각 스테이지가 단순 섬멸전이 아니라 `호위`, `탈출`, `거점 점령`, `버티기`, `일기토 유도` 등 다른 목표를 갖도록 설계한다.
- 삼국지 시대 의상, 깃발, 병종, 무기, 색채 언어가 캐릭터 스프라이트와 이벤트 이미지에 반영되도록 제작 규칙을 정의한다.
- 리소스 제작 파이프라인을 문서화해 AI 기반 에셋 생성 결과물이 바로 게임 구현에 연결되도록 한다.
- `Flutter` 단일 코드베이스로 `Web`과 `Android`를 함께 지원한다.

## 3. Target Audience

- `삼국지 영걸전`, `조조전`, 고전 코에이 SRPG 팬
- 짧고 밀도 있는 턴제 전략 게임을 선호하는 플레이어
- 역사물, 특히 삼국지연의 기반 서사와 장수 관계성에 몰입하는 플레이어
- 모바일과 PC에서 모두 읽기 쉬운 전장 UI를 선호하는 라이트~미들코어 유저

## 4. Product Pillars

### Pillar 1: 유비 중심의 압축된 삼국지 서사

- 도원결의, 반동탁 연합, 서주 구원, 박망파, 장판파, 형주 진입까지 유비 세력의 상승 서사를 따라간다.
- 스토리는 무겁고 진지하지만, 장수 간 충의와 결의를 보여주는 감정 연출을 적극 사용한다.

### Pillar 2: 미션형 SRPG 전장

- 모든 전투는 단순 적 전멸이 아니라 명확한 전술 과제를 가진다.
- 지형, 병과 상성, 병법, 진입 순서, NPC 생존 여부가 결과에 영향을 준다.

### Pillar 3: 소수 정예 장수 육성

- 플레이어가 관리하는 핵심 장수는 5명으로 제한한다.
- 각 장수는 고유 병과 역할, 전용 기술, 일기토 상대, 시그니처 비주얼을 가진다.

### Pillar 4: 삼국지 정체성이 분명한 아트/사운드

- 갑주, 문양, 색상, 무기, 군기, 전장 배경이 한나라 말기와 삼국지연의 이미지를 강하게 환기해야 한다.
- 배경음악은 `고풍 현악 + 전쟁 북소리 + 비장한 멜로디`를 기본 방향으로 한다.

## 6. Core Gameplay Loop

1. 스테이지 브리핑에서 목표, 승리 조건, 패배 조건, 특수 이벤트를 확인한다.
2. 출전 장수 3~5명을 편성하고 장비와 소모품을 조정한다.
3. 실제 플레이 환경에서는 유저가 아군 장수를 직접 선택하고, 이동, 공격, 책략, 도구, 대기 커맨드를 수동으로 입력한다.
4. 지형 보너스와 병과 상성을 활용해 적 주요 장수를 공략한다.
5. 일기토, 구조, 화계, 증원 등 스크립트 이벤트를 처리한다.
6. 전투 후 경험치, 장비, 서사 이벤트를 획득하고 다음 스테이지를 해금한다.

## 7. User Stories

### US-001: 스토리 중심 캠페인 진행

**Description:** 플레이어로서 나는 유비 세력의 입장에서 삼국지 주요 전투를 따라가며 성장하는 서사를 경험하고 싶다.

**Acceptance Criteria:**

- [ ] 캠페인은 10개 스테이지로 구성된다.
- [ ] 각 스테이지는 시작 대화, 전투 중 이벤트, 종료 후 대화가 포함된다.
- [ ] 각 스테이지는 고유 승리 조건과 패배 조건을 가진다.
- [ ] 10개 스테이지의 연결 서사가 도원결의에서 형주 진입까지 이어진다.

### US-002: 턴제 격자 전투

**Description:** 플레이어로서 나는 한 턴씩 유닛을 조작하며 지형과 거리 계산을 통해 전투를 풀고 싶다.

**Acceptance Criteria:**

- [ ] 전장은 정사각 격자 기반이다.
- [ ] 실제 플레이에서는 유저가 아군 유닛을 직접 선택하고, 턴마다 `이동`, `공격`, `책략`, `도구`, `대기` 중 가능한 행동을 수행한다.
- [ ] 공격 가능 범위와 이동 가능 범위가 시각적으로 표시된다.
- [ ] 턴 종료 후 적 AI가 목표 우선순위에 따라 행동한다.
- [ ] 각 스테이지는 최대 턴 수 제한을 가지며, 턴 상한 도달 시 전투가 종료된다.

### US-003: 지형과 병과 상성

**Description:** 플레이어로서 나는 숲, 관문, 강변 같은 지형과 병과 역할을 이용해 열세를 뒤집고 싶다.

**Acceptance Criteria:**

- [ ] 숲, 성문, 도로, 강변, 평지 최소 5종 지형이 존재한다.
- [ ] 지형은 회피율, 이동력, 방어력 중 하나 이상에 영향을 준다.
- [ ] 창병, 보병, 기병, 군사 병과의 차별점이 명확하다.
- [ ] 스테이지 7 `박망파 화계`에서 지형이 전투 승패에 직접 영향을 준다.

### US-004: 일기토와 이벤트 전투

**Description:** 플레이어로서 나는 관우 vs 화웅 같은 상징적 대결을 직접 유도해 원작 감성을 느끼고 싶다.

**Acceptance Criteria:**

- [ ] 최소 4개의 일기토 이벤트가 존재한다.
- [ ] 일기토는 사전에 정의된 특정 장수 조합에서만 발동한다.
- [ ] 일기토는 일반 공격보다 강조된 연출과 별도 결과값을 가진다.
- [ ] 일기토 성공 시 경험치, 사기, 아이템, 이벤트 분기 중 하나의 보상이 제공된다.

### US-005: 소수 정예 장수 육성

**Description:** 플레이어로서 나는 5명의 핵심 장수를 각기 다르게 성장시켜 전투 해법을 바꾸고 싶다.

**Acceptance Criteria:**

- [ ] 플레이어 핵심 장수는 총 5명이다.
- [ ] 각 장수는 고유 패시브 또는 전용 책략 1개 이상을 가진다.
- [ ] 장수는 레벨, 장비, 소모품 슬롯을 가진다.
- [ ] 조운과 제갈량은 합류 전후로 전투 운용 방식이 체감되게 달라진다.

### US-006: 보스 중심 스테이지 설계

**Description:** 플레이어로서 나는 스테이지마다 다른 보스 패턴을 읽고 대응 전략을 세우고 싶다.

**Acceptance Criteria:**

- [ ] 화웅, 여포, 조조, 장료, 하후돈이 최소 1회 이상 중심 적으로 등장한다.
- [ ] 각 보스는 고유 AI 우선순위 또는 전용 기술을 가진다.
- [ ] 여포는 일반 적보다 확실히 높은 위협도를 보여야 한다.
- [ ] 조조는 직접 화력보다 지휘/증원 능력으로 압박해야 한다.

### US-007: PixelLab 기반 캐릭터 스프라이트 제작

**Description:** 아트 제작자로서 나는 삼국지 콘셉트가 반영된 전장용 픽셀 캐릭터를 일관된 규격으로 생성하고 싶다.

**Acceptance Criteria:**

- [ ] 아군 5명, 적군 5명에 대한 기본 스프라이트 생성 규격이 정의된다.
- [ ] 각 캐릭터는 4방향 기준으로 제작된다.
- [ ] `idle`, `walk`, `attack`, `hit` 애니메이션 요구사항이 정의된다.
- [ ] 의상, 무기, 색상 키워드가 캐릭터별로 문서화된다.

### US-008: AI 이미지 기반 배경/이벤트 아트 제작

**Description:** 아트 제작자로서 나는 챕터 일러스트와 전장 배경 이미지를 자동 생성해 제작 속도를 높이고 싶다.

**Acceptance Criteria:**

- [ ] 메인 키아트, 챕터 배너, 스테이지 배경, UI 장식 이미지 목록이 정의된다.
- [ ] 각 이미지 자산별 해상도와 용도가 문서화된다.
- [ ] 이미지 프롬프트는 삼국지 시대감, 군기, 지형, 시간대를 반영해야 한다.
- [ ] 생성 결과는 세계관 톤이 깨지지 않도록 컬러/소품 기준을 따른다.

### US-009: 시대감 있는 음향 패키지

**Description:** 플레이어로서 나는 전투, 메뉴, 승리 순간마다 삼국지 분위기를 강화하는 사운드를 듣고 싶다.

**Acceptance Criteria:**

- [ ] `Gemini BGM`으로 최소 5개 테마 곡 제작 규격이 정의된다.
- [ ] `ElevenLabs`로 최소 10개 효과음 카테고리 제작 규격이 정의된다.
- [ ] 메뉴, 일반 전투, 보스전, 비장 이벤트, 승리 연출에 각기 다른 음향 방향이 존재한다.
- [ ] 모든 오디오는 시대감을 해치지 않는 악기/질감 기준을 따른다.

### US-010: Playwright 기반 화면 검증 루프

**Description:** 개발자로서 나는 화면을 구성한 뒤 실제 렌더링 결과를 확인해 깨진 레이아웃이나 잘못된 표현을 남기지 않고 싶다.

**Acceptance Criteria:**

- [ ] 화면 또는 HUD를 구성한 뒤 반드시 `Playwright`로 실제 렌더링 결과를 확인한다.
- [ ] 확인 시 데스크톱 기준과 필요한 경우 모바일/축소 해상도 기준 표현을 함께 점검한다.
- [ ] 텍스트 잘림, 겹침, 오브젝트 정렬 오류, 타일맵/스프라이트 깨짐, UI 색상 충돌이 있으면 수정 후 재확인한다.
- [ ] `정상 표현`이 확인될 때까지 `구현 -> Playwright 확인 -> 수정 -> 재확인` 루프를 반복한다.

### US-011: 코어 로직과 게임 구현 분리

**Description:** 개발자로서 나는 전투 규칙과 상태 전이를 렌더링/UI와 분리해 구현하여, 실제 게임 화면 없이도 전투를 시뮬레이션하고 검증하고 싶다.

**Acceptance Criteria:**

- [ ] 턴 진행, 이동 판정, 공격 계산, 책략 처리, 승패 판정은 `코어 로직` 계층에서 처리된다.
- [ ] 렌더링, 입력, HUD, 연출, 사운드는 `게임 구현` 계층에서 처리된다.
- [ ] 코어 로직은 화면 없이도 동작 가능한 헤드리스 시뮬레이션 진입점을 제공한다.
- [ ] 동일한 초기 상태와 명령 입력에 대해 시뮬레이션 결과가 재현 가능해야 한다.

### US-012: 화면 목록과 공통 그래픽 규격 선행 선언

**Description:** 개발자와 아트 제작자로서 나는 필요한 화면과 자산 규격을 먼저 고정해, 구현 도중 누락이나 크기 불일치가 생기지 않게 하고 싶다.

**Acceptance Criteria:**

- [ ] 최소 10개 이상의 필수 화면이 문서에 선언되어 있다.
- [ ] 각 화면은 목적과 핵심 UI 구성 요소를 가진다.
- [ ] 캐릭터, 타일맵, 초상화, UI 패널, 배경 이미지의 기준 크기가 문서에 통일되어 있다.
- [ ] 실제 제작 시 임의 크기 사용 대신 정의된 표준 규격을 우선 적용한다.

### US-013: 전투 표시와 남향 아이콘 재사용

**Description:** 개발자와 아트 제작자로서 나는 PixelLab 캐릭터를 전투 화면과 상태/UI 화면에서 일관되게 사용해 자산 낭비 없이 동일한 캐릭터성을 유지하고 싶다.

**Acceptance Criteria:**

- [ ] 전투 화면에서는 각 캐릭터의 `idle`과 `attack` 애니메이션이 실제 표시된다.
- [ ] 캐릭터가 대기 중일 때는 `idle`, 공격 수행 시에는 `attack` 애니메이션을 사용한다.
- [ ] 각 캐릭터의 `남쪽(south)` 방향 기본 프레임을 별도 아이콘 이미지로 추출 또는 재사용한다.
- [ ] 장수 관리 화면과 기타 UI 화면에서는 위 남향 아이콘 이미지를 사용한다.

### US-014: 휴리스틱 AI와 승률 기반 난이도 조정

**Description:** 밸런스 디자이너로서 나는 휴리스틱하게 행동하는 AI를 이용해 스테이지를 반복 시뮬레이션하고, 각 스테이지의 플레이어 승률이 목표치에 수렴하도록 난이도를 조정하고 싶다.

**Acceptance Criteria:**

- [ ] 플레이어측과 적측 모두에 적용 가능한 휴리스틱 AI 정책이 정의된다.
- [ ] 휴리스틱 AI는 목표 우선순위, 생존성, 지형, 사거리, 킬 가능성, 승리 조건 기여도를 반영해 행동을 선택한다.
- [ ] 휴리스틱 AI는 완전 결정론이 아니라, 동점 해소와 패턴 고착 방지를 위한 소량의 랜덤성을 포함한다.
- [ ] 각 스테이지는 헤드리스 시뮬레이션을 통해 충분한 횟수로 반복 검증된다.
- [ ] 시뮬레이션 결과를 바탕으로 `1~3 스테이지는 90%`, `4~10 스테이지는 30%` 플레이어 승률 목표에 맞게 수치와 배치를 조정하는 태스크가 포함된다.

### US-015: 공통 화면 코드와 Playwright 순회 경로

**Description:** 개발자로서 나는 화면별 구현이 제각각 흩어지지 않게 공통 코드를 공유하고, 테스트 가능한 경로를 통해 `Playwright`가 모든 화면을 순회하며 정상 구현 여부를 검증할 수 있게 하고 싶다.

**Acceptance Criteria:**

- [ ] 화면들은 공통 레이아웃, 패널, 버튼, 타이포그래피, 상태 표시 컴포넌트를 공유한다.
- [ ] `Required Screen List`의 각 화면은 최소 1개 이상의 접근 가능한 테스트 경로를 가진다.
- [ ] `Playwright`는 각 화면 경로를 순회하며 렌더링 오류와 레이아웃 깨짐을 검사할 수 있다.
- [ ] 최종 완료 기준에는 `Playwright`의 전체 화면 경로 순회 검증이 포함된다.

## 8. Functional Requirements

- `FR-1`: 게임은 유비 세력 중심의 10개 고정 스테이지 캠페인을 제공해야 한다.
- `FR-1A`: 게임 클라이언트는 `Flutter`로 구현해야 하며, 1차 배포 타겟은 `Web`과 `Android`여야 한다.
- `FR-2`: 플레이어가 직접 육성하고 조작하는 핵심 장수는 정확히 5명이어야 한다.
- `FR-3`: 전투는 턴제 격자 방식이어야 하며, 플레이어 턴과 적 턴이 명확히 구분되어야 한다.
- `FR-4`: 실제 플레이 환경에서는 유저가 아군 유닛을 직접 조작해야 하며, 아군은 `이동`, `공격`, `책략`, `도구`, `대기` 중 허용된 행동을 수행할 수 있어야 한다.
- `FR-5`: 스테이지마다 최소 1개의 특수 목표 또는 이벤트 기믹이 있어야 한다.
- `FR-6`: 전장에는 지형 보너스 시스템이 존재해야 하며, 최소 5종 지형이 구현되어야 한다.
- `FR-7`: 보스 적은 일반 적과 다른 AI 우선순위, 능력치, 연출을 가져야 한다.
- `FR-8`: 최소 4개의 일기토 이벤트가 존재해야 한다.
- `FR-9`: 플레이어는 전투 전 출전 인원, 장비, 소모품을 조정할 수 있어야 한다.
- `FR-10`: 장수는 레벨업을 통해 능력치가 상승하고 일부 전용 기술을 해금해야 한다.
- `FR-11`: 전투 결과에 따라 경험치, 아이템, 스토리 진행도가 지급되어야 한다.
- `FR-12`: 전투 중 패배 조건은 `유비 사망`, `핵심 NPC 사망`, `제한 턴 실패` 중 스테이지별로 명확히 지정되어야 한다.
- `FR-12A`: 모든 스테이지는 명시적인 최대 턴 수를 가져야 하며, 턴 상한 도달 시 즉시 승리 또는 패배 판정을 수행해야 한다.
- `FR-13`: 캐릭터 전장 스프라이트는 `PixelLab` 기준 4방향/픽셀 아트 규격으로 제작되어야 한다.
- `FR-14`: 전장 맵 이미지는 통짜 배경 일러스트가 아니라 `PixelLab`으로 생성한 타일셋을 조합한 타일맵 방식으로 구현되어야 한다.
- `FR-15`: 각 스테이지는 지형 규칙과 연결되는 전용 타일셋 또는 타일 변형 세트를 가져야 한다.
- `FR-16`: 이벤트 일러스트와 UI 장식 이미지는 `Gemini 이미지 MCP`로 생성하는 것을 기본 파이프라인으로 삼아야 한다.
- `FR-17`: 배경음악은 `Gemini BGM`으로, 효과음은 `ElevenLabs`로 생성하는 것을 기본 정책으로 삼아야 한다.
- `FR-18`: 화면/HUD/메뉴/전투 UI를 구성한 뒤 반드시 `Playwright`를 사용해 실제 표현을 검증해야 한다.
- `FR-19`: 시각적 오류가 발견되면 수정 후 `Playwright`로 재검증하는 반복 루프를 정상 표현 시점까지 수행해야 한다.
- `FR-20`: 전투 규칙, 상태 전이, 승패 판정, AI 의사결정은 렌더링/UI 계층과 분리된 독립 코어 모듈에 구현되어야 한다.
- `FR-21`: 코어 모듈은 헤드리스 모드에서 스테이지 시뮬레이션, 리플레이, 테스트 실행이 가능해야 한다.
- `FR-22`: 화면 표현 계층은 코어 상태를 구독/반영하는 어댑터 역할만 수행해야 하며, 전투 규칙을 직접 소유하면 안 된다.
- `FR-23`: 타이틀, 메뉴, 스테이지 선택, 브리핑, 편성, 전투 HUD, 대화, 일기토, 결과, 장수 관리, 저장/불러오기, 설정, 게임 오버 씬이 구현 범위에 포함되어야 한다.
- `FR-24`: 캐릭터 스프라이트, 타일셋, 초상화, UI 패널, 배경 이미지의 기준 제작 크기를 사전에 통일해야 한다.
- `FR-25`: 전투 화면은 각 캐릭터의 `idle` 및 `attack` 애니메이션 상태를 실제로 표시해야 한다.
- `FR-26`: 각 캐릭터의 `south` 방향 기본 프레임은 상태 화면과 기타 UI 화면에서 사용할 아이콘 자산으로 재사용 가능해야 한다.
- `FR-27`: 코어 시뮬레이션에서 동작하는 휴리스틱 AI를 구현해야 한다.
- `FR-28`: 휴리스틱 AI는 공격 우선순위, 위험 회피, 지형 선호, 목표 달성 기여도, 체력 상태를 고려해 행동을 결정해야 한다.
- `FR-28A`: 휴리스틱 AI는 동점 후보 선택, 경미한 이동 변주, 동일 패턴 반복 방지를 위해 통제된 랜덤성을 포함해야 한다.
- `FR-28B`: 휴리스틱 AI는 적군 제어 및 헤드리스 시뮬레이션/밸런싱 검증에 사용하며, 기본 플레이 경험에서 아군 수동 조작을 대체하면 안 된다.
- `FR-29`: 각 스테이지는 휴리스틱 AI 기반 반복 시뮬레이션 이후 `1~3 스테이지 90%`, `4~10 스테이지 30%` 플레이어 승률 목표에 맞춰 난이도를 조정해야 한다.
- `FR-30`: 모든 화면은 공통 레이아웃 및 공유 UI 컴포넌트 체계를 우선 사용해야 한다.
- `FR-31`: `Required Screen List`의 각 화면은 개발/테스트 환경에서 직접 접근 가능한 전용 경로를 가져야 한다.
- `FR-32`: `Playwright`는 위 전용 경로들을 순회하는 방식으로 전체 화면 회귀 검증을 수행해야 한다.
- `FR-33`: 모든 생성형 에셋은 최종 반영 전에 콘셉트 일치성 검수를 거쳐야 한다.

## 9. Design Considerations

### 9.1 Visual Direction

- 기본 톤은 `후한 말 전장 기록화 + 고전 코에이 픽셀 전술 게임`이다.
- UI는 죽간, 비단, 붓글씨, lacquer red, 먹색, 황동 장식에서 모티프를 가져온다.
- 촉한 아군은 `청록/백/황` 계열, 조조군은 `흑청/금`, 여포 세력은 `적흑`으로 색상 언어를 분리한다.
- 얼굴 아이콘과 이벤트 아트는 과도한 판타지 대신 `삼국지연의풍 과장` 정도로 통제한다.

### 9.2 Character Sprite Direction

- 전장 캐릭터는 `PixelLab` 기준 `humanoid`, `n_directions=4`, `view=low top-down`, `size=48 또는 64`를 권장한다.
- `outline=selective outline`, `shading=medium shading`, `detail=medium detail`을 기본값으로 사용한다.
- 갑주와 무기는 장수 실루엣만 봐도 구분되게 설계한다.
- 관우의 장병기, 장비의 굵은 창, 조운의 백은 경갑, 제갈량의 부채는 전장에서 즉시 식별 가능해야 한다.

### 9.3 Stage Art Direction

- 10개 스테이지는 각각 시각적 장소성이 뚜렷해야 한다.
- 예시: 사수관의 먼지 낀 관문, 호로관의 붉은 노을, 박망파의 숲길, 장판파의 난민 행렬, 형주의 강변 성채.
- 전장 맵은 한 장의 배경 그림이 아니라 `PixelLab` 타일셋을 조합한 타일맵으로 구축한다.
- 배경 타일은 과도한 현실감보다 읽기 쉬운 전략 가독성을 우선한다.
- 숲, 관문, 성벽, 도로, 강변, 평지 타일은 지형 효과를 시각적으로 즉시 구분할 수 있어야 한다.

### 9.4 Screen Composition Policy

- 모든 화면은 먼저 `Required Screen List`에 정의된 목적을 기준으로 설계한다.
- 전투 외 화면도 동일한 삼국지 시각 언어를 유지해야 한다.
- 화면마다 임시 레이아웃을 새로 만들기보다 공통 패널, 버튼, 타이포 스케일을 재사용한다.
- 전투 HUD와 대화 화면은 가장 사용 빈도가 높으므로 가독성과 정보 계층을 최우선으로 한다.
- 화면 구현은 페이지별 개별 하드코딩보다 공유 레이아웃과 재사용 컴포넌트를 우선해야 한다.
- 테스트와 검증을 위해 각 화면은 독립 진입 가능한 라우트 또는 테스트 경로를 제공해야 한다.

## 10. Technical Considerations

### 10.0 Implementation Platform

- 클라이언트 구현 기술은 `Flutter`로 고정한다.
- 1차 지원 플랫폼은 `Web`과 `Android`다.
- 화면, 입력, UI 상태, 라우팅, 애니메이션 표현 계층은 `Flutter` 기준으로 설계한다.
- 코어 로직은 `Flutter` UI 계층과 분리하되, 실제 런타임 어댑터는 `Flutter`에서 연결한다.

  - 초상화 기본 크기: `256x256`

### 10.2 Gameplay System Constraints

- 캠페인 범위가 10개 전투로 제한되어 있으므로 분기 시나리오는 최소화한다.
- 육성 장수가 5명뿐이므로 각 장수의 전투 역할 중복을 피해야 한다.
- 스테이지 수가 적은 대신 각 전투는 확실한 기억 포인트를 가져야 한다.
- 맵 제작은 전장 1장 단위의 페인팅보다 타일셋 재사용을 우선해야 하므로, 스테이지 테마별 공용 타일 라이브러리 설계가 필요하다.
- AI 생성 자산은 스타일 편차가 발생할 수 있으므로 프롬프트 템플릿과 검수 체크리스트가 반드시 필요하다.

### 10.3 Core Logic Architecture

- 게임 구현은 `Core Logic`과 `Game Runtime` 두 층으로 분리한다.
- `Core Logic` 책임:
  - 턴 순서 관리
  - 최대 턴 수 카운트 및 제한 판정
  - 이동 가능 범위 계산
  - 전투/책략 수치 계산
  - 지형 효과 적용
  - 상태 이상, 버프/디버프 처리
  - 승리/패배 조건 판정
  - 적 AI 의사결정
- `Game Runtime` 책임:
  - 렌더링
  - 입력 처리
  - 카메라/HUD
  - 애니메이션 연출
  - 오디오 재생
  - 컷신 및 화면 전환
- 코어 로직은 특정 엔진 UI 객체, 렌더러, 사운드 시스템, 비동기 연출 타이밍에 의존하면 안 된다.
- 코어 로직은 `현재 게임 상태 + 플레이어/AI 명령 -> 다음 게임 상태` 형태로 계산 가능한 구조를 가져야 한다.
- 코어 상태는 직렬화 가능해야 하며, 저장/불러오기와 리플레이 입력 재생에 활용할 수 있어야 한다.
- 스테이지 데이터, 유닛 스탯, 스킬 정의는 가능하면 데이터 중심 구조로 분리해 시뮬레이션과 런타임이 동일 데이터를 공유하게 해야 한다.

### 10.4 Simulation and Verification

- 코어 로직은 별도 헤드리스 실행 경로를 제공해야 한다.
- 헤드리스 시뮬레이션에서 가능한 작업:
  - 특정 스테이지 초기화
  - 미리 정의한 명령 시퀀스 재생
  - AI vs AI 자동 전투
  - 승패 조건 검증
  - 밸런스/회귀 테스트
- 시뮬레이션 결과는 최소한 다음 정보를 출력 가능해야 한다.
  - 턴 수
  - 최대 턴 수 도달 여부
  - 생존 유닛 목록
  - 승리/패배 원인
  - 핵심 이벤트 발동 여부
- 전투 규칙 수정 시 UI를 먼저 만지는 방식이 아니라, 코어 시뮬레이션으로 규칙 검증 후 런타임 표현에 연결하는 절차를 권장한다.

### 10.5 Heuristic AI and Balance Simulation

- 시뮬레이션용 AI는 학습형 모델이 아니라 규칙 기반 휴리스틱으로 구현한다.
- 휴리스틱 AI는 기본적으로 규칙 기반이지만, 완전 고정 패턴이 되지 않도록 소량의 랜덤성을 포함한다.
- 기본 휴리스틱 평가 축:
  - 현재 턴에 처치 가능한 대상 우선
  - 승리 조건에 직접 기여하는 위치/행동 우선
  - 반격 위험이 낮은 선택 우선
  - 방어 지형 점유 선호
  - 저체력 유닛의 후퇴 또는 보호 우선
  - 보스/핵심 장수 생존 가치 반영
- 행동 선택 예시:
  - 공격 가능 시 `즉시 킬 > 핵심 목표 공격 > 반격 리스크 낮은 공격 > 지형 유리 위치 선점`
  - 이동만 가능 시 `목표 지점 접근 > 방어 지형 확보 > 아군 집결 > 위험 회피`
  - 책략 가능 시 `승리 조건 기여도와 기대 효율이 물리 공격보다 높은 경우 우선 사용`
- 랜덤성 적용 원칙:
  - 최고 점수 행동이 여러 개일 경우, 그중 하나를 확률적으로 선택할 수 있다.
  - 점수 차이가 매우 작은 후보들 사이에서는 낮은 강도의 무작위 편차를 허용한다.
  - 치명적인 오판을 유도하는 과도한 랜덤성은 금지하며, 휴리스틱 우선순위가 항상 기본 골격이 되어야 한다.
  - 시뮬레이션 재현이 필요할 때는 난수 시드를 고정할 수 있어야 한다.
- 밸런싱 절차:
  - 각 스테이지를 동일 조건에서 반복 시뮬레이션한다.
  - 플레이어측 휴리스틱 AI vs 적측 휴리스틱 AI 기준으로 결과를 수집한다.
  - `1~3 스테이지`는 플레이어 승률 목표를 `90%`로 설정한다.
  - `4~10 스테이지`는 플레이어 승률 목표를 `30%`로 설정한다.
  - 목표 승률보다 높으면 적 증원, 능력치, 지형 불리, 턴 제한 강화 등을 검토한다.
  - 목표 승률보다 낮으면 적 능력치 완화, 출전 조건 완화, 아이템 제공, 지형 이점 추가 등을 검토한다.
  - 조정 후 다시 시뮬레이션하여 목표 승률 범위에 수렴할 때까지 반복한다.
- 권장 측정 방식:
  - 스테이지당 최소 `100회` 이상 반복 시뮬레이션
  - 승률, 평균 턴 수, 주요 장수 생존율, 특정 이벤트 발동률 함께 기록
- 주의:
  - 초반 3개 스테이지는 튜토리얼/적응 구간으로 간주해 높은 승률을 유지한다.
  - 4스테이지 이후부터는 본격 난도 구간으로 전환한다.

### 10.6 Visual QA and Verification

- 검증 대상:
  - 메인 메뉴
  - 스테이지 선택 화면
  - 전투 HUD
  - 스토리 대화 화면
  - 결과 화면
  - 그 외 `Required Screen List`에 선언된 나머지 화면 전체
- 확인 항목:
  - 텍스트 잘림 여부
  - 버튼/패널 겹침 여부
  - 스프라이트와 타일맵 정렬 상태
  - 색 대비와 가독성
  - 해상도 변화 시 레이아웃 안정성
- 권장 검증 방식:
  - 화면별 테스트 전용 경로 준비
  - `Playwright`가 해당 경로를 순차 방문
  - 각 경로에서 필수 UI 요소 존재 여부 확인
  - 스크린샷 또는 렌더 결과 비교를 통해 이상 여부 확인
- 반복 원칙:
  - `구현`
  - `Playwright 확인`
  - `문제 기록`
  - `수정`
  - `재확인`
- 위 루프는 `정상 표현`이 확인될 때까지 반복해야 하며, 수동 육안 추정만으로 완료 처리할 수 없다.


### 10.7 Engineering Baseline Review

- `2026-03-29` 기준 저장소는 `Flutter` 기본 카운터 앱을 벗어나, PRD 핵심 축을 실제 코드와 테스트로 검증할 수 있는 SRPG 수직 슬라이스 베이스라인을 갖췄다. 현재 구현은 아래 책임 분리를 따른다.
  - `lib/src/app/`: 공통 테마와 `SC-01 ~ SC-14` 전체 화면을 순회할 수 있는 라우팅 셸
  - `lib/src/data/game_data.dart`: 10개 스테이지, 5명 아군, 5명 네임드 적장을 담은 정적 데이터 저장소
  - `lib/src/domain/`: `BattleEngine`, 전장 모델, 헤드리스 시뮬레이션 진입점
  - `lib/src/presentation/screens.dart`: 타이틀, 메뉴, 스테이지 선택, 브리핑, 전투 HUD 등 주요 화면 목업
  - `lib/app/`, `lib/core/`, `lib/content/`: 기존 import 경로를 유지하기 위한 얇은 re-export 호환 계층
- 즉, 현재 단계는 `완전한 초기 스캐폴드`가 아니라 `M1(스캐폴드 교체)`와 `M2(코어 전투 엔진)`의 초반 품질 게이트를 이미 통과할 수 있는 상태에 가깝다.
- 이번 리뷰에서 확인된 강점:
  - `Required Screen List`에 대응하는 14개 경로가 앱에 선언되어 있어 화면 순회용 허브로 활용 가능하다.
  - 전장 데이터가 화면 코드와 분리되어 있어 스테이지/장수 정의를 반복적으로 교체·확장하기 쉽다.
  - `BattleEngine.simulate()`와 수동 이동/공격/대기 커맨드가 존재해 헤드리스 검증 방향 자체는 PRD와 일치한다.
  - `flutter analyze`, `flutter test`가 모두 통과하며, 전투 엔진/라우트/HUD 회귀 테스트가 현재 수직 슬라이스를 보호한다.
- 이번 리뷰에서 확인된 우선 품질 리스크:
  - `lib/src/presentation/screens.dart` 한 파일이 14개 화면을 모두 소유하고 있어 이후 기능 추가 시 결합도와 리뷰 범위가 빠르게 커질 가능성이 높다.
  - `lib/src/domain/models.dart`가 `Color` 표현을 위해 `package:flutter/material.dart`를 import 하므로, `10.8`이 요구하는 `순수 Dart 코어` 경계는 아직 완전히 달성되지 않았다.
  - `Playwright` 기반 웹 라우트 순회 및 시각 검증 스크립트는 아직 저장소에 없으므로 `US-010`, `US-015`, `10.6`의 검증 체인은 미구현 상태다.
  - `lib/src/data/game_data.dart` 한 파일에 캠페인/장수/화면 정의가 모두 집중돼 있어 콘텐츠 스케일업 시 파일 분리 전략이 필요하다.
- 검증 스냅샷(리뷰 시점):
  - `flutter analyze` → `PASS` (`No issues found!`)
  - `flutter test` → `PASS` (`All tests passed!`)
- 따라서 다음 구현 우선순위는 단순 콘텐츠 추가보다 아래 3가지를 병행하는 것이 바람직하다.
  - `screens.dart`를 전투/캠페인/관리 화면 단위로 분리해 UI 결합도를 낮춘다.
  - `domain` 계층에서 Flutter UI 타입 의존을 제거해 PRD가 요구한 순수 Dart 코어 경계를 복구한다.
  - `Playwright` 기반 라우트 순회/시각 검증 체인을 추가해 문서상의 품질 게이트를 실제 자동화와 맞춘다.

### 10.8 Recommended Project Structure

- 현재 기준 합의 가능한 디렉터리 책임 분리:
  - `lib/src/app/`: 앱 부트스트랩, 라우팅, 테마, 공통 레이아웃
  - `lib/src/domain/`: 전투 규칙, 상태 전이, 시뮬레이션, 데이터 모델
  - `lib/src/data/`: 스테이지/장수/화면 정적 데이터와 저장소
  - `lib/src/presentation/`: 수직 슬라이스 UI 화면과 프리젠테이션 레이어
  - `lib/app/`, `lib/core/`, `lib/content/`: 외부/테스트 import 호환을 위한 얇은 re-export 계층
  - `test/core/`: 이동/전투/승패/시뮬레이션 회귀 테스트
  - `test/features/`: 위젯 및 상태 연결 테스트
  - `tooling/playwright/` 또는 동등 디렉터리: 웹 라우트 순회 및 시각 검증 스크립트
- 중기 확장 목표(콘텐츠/기능이 늘어날 때):
  - `lib/features/campaign/`: 스테이지 선택, 브리핑, 결과, 저장 흐름
  - `lib/features/battle/`: 전투 HUD, 타일맵 렌더링, 입력 처리, 컷인/연출 어댑터
  - `lib/features/roster/`: 장수 관리, 편성, 장비 UI
  - `assets/`: 스프라이트, 타일셋, 초상화, 이벤트 이미지, 오디오를 표준 규격별로 정리
- 구조 원칙:
  - `lib/src/domain/`은 최종적으로 `flutter` UI 패키지 의존 없이 테스트 가능해야 한다.
  - `lib/src/presentation/` 또는 향후 `lib/features/*`는 직접 수치 계산을 소유하지 않고 `domain/core`에 위임해야 한다.
  - `lib/src/data/` 데이터는 시뮬레이션과 실제 런타임에서 동일하게 참조해야 한다.
  - `lib/app/`, `lib/core/`, `lib/content/`는 기능을 직접 소유하지 않고 re-export만 유지해야 한다.
  - 화면 전용 더미 데이터가 필요해도 최종 데이터 모델과 동일한 스키마를 써야 한다.

  - 코어 규칙 변경 시 시뮬레이션 회귀 테스트 또는 동등한 재현 가능한 검증 기록이 함께 갱신되어야 한다.

## 12. Success Metrics

- 첫 캠페인 완주 시간이 `2~3시간` 범위에 들어온다.
- 플레이테스트 참가자의 70% 이상이 `각 스테이지 목표가 다르게 느껴진다`고 응답한다.
- 플레이테스트 참가자의 70% 이상이 `유비/관우/장비/조운/제갈량의 역할 차이가 명확하다`고 응답한다.
- 주요 시각 자산 검수에서 `삼국지 시대감이 충분하다` 평가가 80% 이상 나온다.
- 생성형 에셋 재작업률이 30% 이하로 유지된다.
- `Playwright` 검증 기준에서 치명적 UI 깨짐 이슈가 릴리즈 시점에 0건이어야 한다.
- 주요 전투 규칙 변경 시 헤드리스 시뮬레이션 회귀 검증이 가능해야 한다.
- 밸런싱 기준 시뮬레이션에서 `1~3 스테이지는 90%`, `4~10 스테이지는 30%` 플레이어 승률 목표에 수렴해야 한다.
- `Required Screen List`의 모든 화면 경로가 `Playwright` 순회 검증을 통과해야 한다.
